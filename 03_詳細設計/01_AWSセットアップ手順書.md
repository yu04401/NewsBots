## 1. 全体の流れ

```
STEP 1  AWSアカウント登録
   ↓
STEP 2  rootユーザー MFA設定（セキュリティ）
   ↓
STEP 3  請求アラート設定（意図しない課金防止）
   ↓
STEP 4  IAM Identity Center セットアップ（作業用ユーザー作成）
   ↓
STEP 5  SSM Parameter Store にシークレット登録
        - /newsbot/anthropic_api_key
        - /newsbot/discord_webhook_url
        - /newsbot/discord_channel_name
   ↓
STEP 6  Lambda 関数を作成
   ↓
STEP 7  Lambda の IAM ロールに SSM 読み取り権限を付与
   ↓
STEP 8  EventBridge Scheduler で毎朝6時トリガーを設定
   ↓
STEP 9  手動テスト実行 → Discord への配信を確認
```

> **作業前に準備するもの**
> - クレジットカード（登録必須。無料枠内なら請求なし）
> - 携帯電話番号（SMS認証用。IP電話は不可）
> - メールアドレス（rootユーザー用。使い捨て不可）
> - Anthropic APIキー（[console.anthropic.com](https://console.anthropic.com) で事前取得）
> - Discord Webhook URL（配信チャンネルで事前取得）

## 2. AWSアカウント登録

### 2.1 登録ページへアクセス

`https://portal.aws.amazon.com/billing/signup` にアクセスする。

### 2.2 登録ステップ

以下の順でフォームを入力する。

| ステップ | 入力内容 | 注意 |
|---------|---------|------|
| ① メールアドレス・アカウント名 | rootユーザーのメールアドレスとアカウント名を入力 → 確認コードがメールに届く | アカウント名は後から変更可能 |
| ② パスワード設定 | rootユーザーのパスワードを設定 | 英数字記号を含む強固なパスワードを使用 |
| ③ 連絡先情報 | 個人 or ビジネスを選択 → 住所・電話番号を入力 | 電話番号はSMS受信できるものを使用 |
| ④ 電話番号確認 | SMSまたは音声通話で確認コードを受信・入力 | IP電話（050番号等）は不可 |
| ⑤ クレジットカード登録 | カード情報を入力（無料枠内は請求なし） | $1の仮払いが発生する場合があるが自動返金される |
| ⑥ サポートプラン選択 | **「Basic support - Free」** を選択 | 個人・小規模利用はBasicで十分 |
| ⑦ 完了 | 登録完了メールを確認 | アクティベーションに最大数時間かかる場合あり |

### 2.3 登録後の確認

1. `https://console.aws.amazon.com` にrootユーザーのメールアドレスとパスワードでサインイン
2. 右上のリージョンが **「アジアパシフィック（東京） ap-northeast-1」** になっていることを確認
3. なっていない場合はリージョンドロップダウンから「アジアパシフィック（東京）」を選択

## 3. rootユーザーのセキュリティ設定

> rootユーザーは全権限を持つ特別なアカウント。MFAを設定した後は**日常作業では使用しない**。

### 3.1 MFA（多要素認証）の設定

1. AWSコンソール右上のアカウント名 → **「セキュリティ認証情報」** をクリック
2. 「多要素認証(MFA)」セクション → **「MFAを割り当てる」** をクリック
3. MFAデバイスの選択:

| 種別 | 推奨度 | 説明 |
|------|-------|------|
| 認証アプリ（仮想MFA） | **推奨** | Google Authenticator / Authy など。スマートフォンで利用可能 |
| ハードウェアキー（FIDO2） | 最高セキュリティ | YubiKey等。コストがかかる |
| ハードウェアTOTP | - | 専用トークンデバイス |

4. 「認証アプリ」を選択 → QRコードをスマートフォンのAuthenticatorアプリでスキャン
5. アプリに表示される6桁のコードを2回（30秒ごとに切り替わる番号）入力して完了

### 3.2 rootユーザーのアクセスキーを削除

1. 「セキュリティ認証情報」→「アクセスキー」セクションを確認
2. アクセスキーが存在する場合は**削除する**（rootのアクセスキーは不要かつ危険）

## 4. 請求アラート設定

意図しない課金を早期検知するために **AWS Budgets** を設定する。

### 4.1 IAMユーザーへの請求情報アクセス許可（rootで実施）

> この設定はrootユーザーでのみ変更可能。

1. AWSコンソール右上のアカウント名 → **「アカウント」** をクリック
2. 「IAMユーザーおよびロールによる請求情報へのアクセス」セクションまでスクロール
3. 「編集」→ **「IAMアクセスのアクティブ化」** にチェック → 「更新」

### 4.2 AWS Budgets でアラートを作成

1. AWSコンソール検索バーに「Budgets」と入力 → **「Budgets」** を開く
2. **「予算を作成」** をクリック
3. 設定内容:

| 設定項目 | 値 |
|---------|---|
| 予算タイプ | コスト予算 |
| 予算名 | `newsbot-monthly-budget` |
| 予算期間 | 月次 |
| 予算額 | `5` USD（余裕を持って設定） |

4. アラートしきい値を **2段階** で設定:

| アラート | しきい値 | 通知先 |
|---------|--------|-------|
| アラート1 | 実績コストが予算の **80%** を超えた場合 | 自分のメールアドレス |
| アラート2 | 実績コストが予算の **100%** を超えた場合 | 自分のメールアドレス |

5. 「予算を作成」をクリック
6. 通知先メールアドレスに確認メールが届いたら **「Confirm subscription」** をクリック

> **補足**: このプロジェクトのAWS費用は月$0（無料枠内）。Claude APIのみ月$0.05〜$0.15程度のため、$5のアラートが発動することはまずないが、誤設定による意図しない課金の保険として設定しておく。

## 5. IAM Identity Centerのセットアップ

> 2026年現在、AWSは人間のアクセスに **IAM Identity Center（SSO）** の利用を強く推奨している。IAMユーザーの新規作成は非推奨。

### 5.1 IAM Identity Centerを有効化

1. AWSコンソール検索バーに「IAM Identity Center」と入力 → 開く
2. **「有効にする」** をクリック
3. 「AWS Organizations で有効にする」または「このアカウントで有効にする」を選択
   - 単一アカウントの場合: **「このアカウントのみ有効にする（アカウントインスタンス）」** を選択
4. 有効化が完了するまで待機（数十秒）

### 5.2 ユーザーを作成

1. 左メニュー **「ユーザー」** → **「ユーザーを追加」**
2. 以下を入力:

| 項目 | 値 |
|-----|---|
| ユーザー名 | `admin` （任意） |
| メールアドレス | 管理者のメールアドレス（rootとは別のメール推奨） |
| 名・姓 | 任意 |

3. グループは後で設定するためスキップしてもよい → 「ユーザーを追加」
4. 招待メールが届いたら、リンクからパスワードを設定してサインインを確認

### 5.3 許可セットを作成

1. 左メニュー **「許可セット」** → **「許可セットを作成」**
2. 「事前定義済みの許可セット」→ **「AdministratorAccess」** を選択 → 「次へ」
3. 許可セット名: `AdministratorAccess`（デフォルトのまま） → 「作成」

### 5.4 ユーザーとAWSアカウントを紐付ける

1. 左メニュー **「AWSアカウント」** → 対象のアカウントにチェック → **「ユーザーまたはグループを割り当て」**
2. 先ほど作成したユーザーを選択 → 「次へ」
3. 許可セット「AdministratorAccess」を選択 → 「送信」

### 5.5 IAM Identity Center でのサインイン

1. 左メニュー「設定」→「アイデンティティソース」内の **「AWSアクセスポータルURL」** をコピー
   - 例: `https://d-xxxxxxxxxx.awsapps.com/start`
2. このURLをブックマークする
3. 以降はこのURLから作業用ユーザーでサインインして作業を行う（**rootユーザーは使わない**）

## 6. SSM Parameter Storeへのシークレット登録

> ここからの作業は IAM Identity Center のポータルから作業用ユーザーでサインインして実施する。

### 6.1 登録するパラメータ一覧

| パラメータ名 | タイプ | 値 |
|------------|-------|---|
| `/newsbot/anthropic_api_key` | SecureString | Anthropic APIキー |
| `/newsbot/discord_webhook_url` | SecureString | Discord Webhook URL |
| `/newsbot/discord_channel_name` | String | 配信チャンネル名（例: `it-news`） |

### 6.2 登録手順（コンソール）

1. AWSコンソール検索バーに「Systems Manager」と入力 → 開く
2. 左メニュー **「パラメータストア」** をクリック
3. **「パラメータを作成」** をクリック
4. 以下の設定で各パラメータを作成:

#### `/newsbot/anthropic_api_key`

| 設定項目 | 値 |
|---------|---|
| 名前 | `/newsbot/anthropic_api_key` |
| 説明 | Anthropic Claude API キー |
| 階層 | 標準 |
| タイプ | **SecureString** |
| KMSキーソース | 現在のアカウント → `alias/aws/ssm`（デフォルト） |
| 値 | Anthropic APIキーの値を貼り付け |

「パラメータを作成」をクリック。

#### `/newsbot/discord_webhook_url`

| 設定項目 | 値 |
|---------|---|
| 名前 | `/newsbot/discord_webhook_url` |
| 説明 | Discord Webhook URL（配信チャンネルに紐付き） |
| 階層 | 標準 |
| タイプ | **SecureString** |
| KMSキーソース | 現在のアカウント → `alias/aws/ssm`（デフォルト） |
| 値 | Discord Webhook URLを貼り付け |

「パラメータを作成」をクリック。

#### `/newsbot/discord_channel_name`

| 設定項目 | 値 |
|---------|---|
| 名前 | `/newsbot/discord_channel_name` |
| 説明 | 配信先Discordチャンネル名（ログ確認用） |
| 階層 | 標準 |
| タイプ | **String** |
| 値 | チャンネル名（例: `it-news`） |

「パラメータを作成」をクリック。

### 6.3 登録確認

パラメータストアの一覧画面で以下の3件が表示されていることを確認する。

```
/newsbot/anthropic_api_key     SecureString  標準
/newsbot/discord_webhook_url   SecureString  標準
/newsbot/discord_channel_name  String        標準
```

## 7. Lambda関数の作成

### 7.1 Lambda関数を新規作成

1. AWSコンソール検索バーに「Lambda」と入力 → 開く
2. リージョンが **「東京（ap-northeast-1）」** であることを確認
3. **「関数の作成」** をクリック
4. 以下の設定で作成:

| 設定項目 | 値 |
|---------|---|
| 作成方法 | **一から作成** |
| 関数名 | `newsbot-delivery` |
| ランタイム | **Python 3.13** |
| アーキテクチャ | x86_64 |
| 実行ロール | 「基本的なLambdaアクセス権限で新しいロールを作成」 |

5. **「関数の作成」** をクリック

> **Python 3.13について**: 要件定義書はPython 3.12を記載しているが、2026年現在の推奨はPython 3.13（長期サポート対象。2029年10月まで保証）のため3.13を採用する。動作に差異はない。

### 7.2 タイムアウトとメモリの設定変更

デフォルト設定では処理が途中で打ち切られる可能性があるため変更する。

1. 関数ページ → **「設定」タブ** → **「一般設定」** → 「編集」
2. 以下に変更して「保存」:

| 設定項目 | デフォルト | 変更後 | 理由 |
|---------|---------|-------|------|
| タイムアウト | 3秒 | **5分（300秒）** | RSS取得・Claude API呼び出しを含むため |
| メモリ | 128 MB | **256 MB** | 複数ソース取得時のメモリ確保 |

### 7.3 コードのデプロイ（暫定）

実装フェーズでコードを用意するまでの暫定として、関数が正常に起動するか確認できる最低限のコードを配置する。

1. 関数ページ → **「コード」タブ**
2. コードエディタに以下を貼り付けて「Deploy」をクリック:

```python
import json

def lambda_handler(event, context):
    print("[START] NewsBot Lambda - placeholder")
    return {
        'statusCode': 200,
        'body': json.dumps('Placeholder OK')
    }
```

## 8. LambdaのIAMロール設定

Lambda関数がSSM Parameter Storeを読み取れるよう、実行ロールに権限を追加する。

### 8.1 実行ロールを開く

1. Lambda関数ページ → **「設定」タブ** → **「アクセス権限」**
2. 「実行ロール」のロール名リンク（例: `newsbot-delivery-role-xxxx`）をクリック
   - IAMコンソールにリダイレクトされる

### 8.2 インラインポリシーを追加

1. IAMコンソールのロール詳細ページ → **「許可を追加」** → **「インラインポリシーを作成」**
2. 「JSON」タブを選択し、以下を貼り付ける:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "SSMReadNewsbot",
      "Effect": "Allow",
      "Action": [
        "ssm:GetParameter",
        "ssm:GetParameters"
      ],
      "Resource": "arn:aws:ssm:ap-northeast-1:*:parameter/newsbot/*"
    }
  ]
}
```

3. **「次へ」** → ポリシー名に `NewsBot-SSMReadPolicy` と入力 → **「ポリシーを作成」**

### 8.3 権限の確認

ロールの許可ポリシー一覧に以下の2つが表示されていることを確認:

| ポリシー名 | 種別 | 目的 |
|----------|-----|------|
| `AWSLambdaBasicExecutionRole-xxxxx` | AWS管理ポリシー | CloudWatch Logsへのログ出力 |
| `NewsBot-SSMReadPolicy` | インラインポリシー | SSM Parameter Storeの読み取り |

### 8.4 SSMアクセスの動作テスト

1. Lambda関数のコードエディタで以下のテストコードに差し替えて「Deploy」:

```python
import boto3
import json

def lambda_handler(event, context):
    ssm = boto3.client('ssm', region_name='ap-northeast-1')

    # SSMから読み取りテスト
    channel = ssm.get_parameter(
        Name='/newsbot/discord_channel_name'
    )['Parameter']['Value']

    print(f"[SSM] channel_name = {channel}")
    return {'statusCode': 200, 'body': json.dumps(f'channel: {channel}')}
```

2. 「テスト」タブ → 「テストイベントを作成」→ イベント名 `test` → `{}` のまま「保存」→ **「テスト」**
3. 実行結果に `"channel: it-news"` 等が表示されれば SSM 読み取り権限の設定は完了

> テスト後は暫定コードに戻しておく（実装フェーズで本番コードに差し替える）。

## 9. EventBridge Schedulerの設定

毎朝6:00 JST に Lambda を自動起動するスケジュールを設定する。

> **注意**: 2026年現在、新規設定には「EventBridge Scheduler」を使用する。AWSコンソールの「EventBridge Rules」は「(レガシー)」と表記されており非推奨。

### 9.1 スケジュールを作成

1. AWSコンソール検索バーに「EventBridge」と入力 → 開く
2. 左メニュー **「スケジューラ」** → **「スケジュール」** → **「スケジュールの作成」**

### 9.2 スケジュールの詳細設定

**「スケジュールの詳細を指定する」**

| 設定項目 | 値 |
|---------|---|
| スケジュール名 | `newsbot-daily-6am` |
| スケジュールグループ | `default` |

**「スケジュールのパターン」**

| 設定項目 | 値 |
|---------|---|
| 繰り返しスケジュール | **Cron ベースのスケジュール** |
| Cron式 | `0 21 * * ? *` |
| タイムゾーン | **Asia/Tokyo** |
| フレキシブルな時間枠 | **オフ**（時刻を厳守） |

> **Cron式の解説**
> ```
> cron(分 時 日 月 曜日 年)
>      0  21  *  *   ?   *
> ```
> UTC 21:00 = JST 翌06:00。「日」と「曜日」は同時指定不可のため曜日を `?` にする。

**次へ**

### 9.3 ターゲットの設定

| 設定項目 | 値 |
|---------|---|
| ターゲットAPI | **テンプレートされたターゲット** |
| ターゲット | **AWS Lambda - Invoke** |
| Lambda関数 | `newsbot-delivery` |
| バージョン/エイリアス | `$LATEST` |
| ペイロード | 空欄でよい |

**次へ**

### 9.4 設定

| 設定項目 | 値 |
|---------|---|
| スケジュールの状態 | 有効 |
| アクションの再試行 | 最大試行回数: `2`、最大期間: `1時間` |
| 実行ロール | **「このスケジュール用の新しいロールを作成する」** |
| デッドレターキュー | なし |

「スケジュールを作成」をクリック。

### 9.5 設定確認

スケジュール一覧に `newsbot-daily-6am` が表示され、ステータスが **「有効」** になっていることを確認する。

## 10. 動作確認

### 10.1 Lambdaの手動テスト実行

本番コード（実装フェーズで用意）をデプロイした後、以下の手順で手動テストを実施する。

1. Lambda関数ページ → **「テスト」タブ**
2. イベント名 `manual-test`、ペイロード `{}` でテストを実行
3. 確認ポイント:

| 確認項目 | 期待する結果 |
|---------|-----------|
| 実行ステータス | 成功（緑のチェックマーク） |
| CloudWatch Logs | `[START]` → `[FETCH]` → `[AI]` → `[DISCORD]` → `[END]` の順でログが出力されている |
| Discordチャンネル | 指定チャンネルに5件のEmbedメッセージが投稿されている |

### 10.2 CloudWatch Logsの確認方法

1. Lambda関数ページ → **「モニタリング」タブ** → **「CloudWatch のログを表示」**
2. 最新のログストリームをクリック
3. ログイベントを確認し、エラーがないことを検証

### 10.3 よくある初回エラーと対処法

| エラー内容 | 原因 | 対処法 |
|-----------|-----|-------|
| `AccessDeniedException: ... ssm:GetParameter` | IAMロールのSSM権限が不足 | [8.2](#82-インラインポリシーを追加) を再確認 |
| `Task timed out after 300.00 seconds` | 処理が5分以内に完了しない | タイムアウト値を延ばすか、コードの処理を最適化 |
| `ParameterNotFound: /newsbot/...` | SSMのパラメータ名が不一致 | [6.3](#63-登録確認) でパラメータ名を確認 |
| Discordにメッセージが届かない | Webhook URLが無効 | Discord側でWebhookが削除されていないか確認 |
| `NewConnectionError` (TechCrunch等) | CloudflareによるIPブロック | そのソースをスキップして他ソースで継続する実装に変更 |
