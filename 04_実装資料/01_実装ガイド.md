## 2. 各モジュールの責務と主要ロジック

### config.py

**責務:** 定数・設定値の一元管理

SSMパラメータ名、AWS リージョン、Claude モデル名、ニュース取得設定、Discord 設定、6つのニュースソース定義をすべてここで管理する。コードの他の場所でマジックナンバーや文字列リテラルを使わない。

### models.py

**責務:** アプリケーション全体で使うデータ構造の定義

`@dataclass(slots=True)` を使用して2つのデータクラスを定義する。`slots=True` により通常のdataclassより軽量になる。

- `Article`: ニュース記事1件（title, url, source, color, summary, rank）
- `BotConfig`: SSMから取得したシークレット一式（anthropic_api_key, discord_webhook_url, discord_channel_name）

### ssm_client.py

**責務:** AWS SSM Parameter Store からシークレットを取得して BotConfig を返す

**ローカルテスト用フォールバック:** `LOCAL_TEST=1` 環境変数が設定されている場合は SSM を呼び出さず、環境変数から直接読み込む。

```python
# ローカル実行の場合の動作
if os.environ.get("LOCAL_TEST") == "1":
    return BotConfig(
        anthropic_api_key=os.environ["ANTHROPIC_API_KEY"],
        ...
    )
```

### news_fetcher.py

**責務:** 複数のニュースソースから記事を取得し、URL重複を除去した Article リストを返す

**主要ロジック:**

- `fetch_all_articles()`: `NEWS_SOURCES` をループし、RSS と Hacker News で分岐
- URL の重複除去は `seen_urls: set[str]` で管理（同じ URL が複数ソースに出ても1件のみ保持）
- 1ソースが失敗しても `WARNING` ログを出力して他のソースの処理を継続
- `_fetch_rss()`: `feedparser.parse()` を使用。`bozo=True` かつ `entries` が空の場合はリトライ
- `_fetch_hackernews()`: まず `topstories.json` でID一覧を取得し、各IDの詳細を順次リクエスト

### ai_ranker.py

**責務:** Claude AI（claude-haiku-4-5）に記事リストを渡し、トップ5件を厳選・日本語要約させる

**主要ロジック:**

システムプロンプトで JSON 配列のみを出力するよう指示する。Claude のレスポンスは以下の形式：

```json
[
  {"index": 2, "rank": 1, "summary": "日本語要約文"},
  ...
]
```

`index` は渡した記事リストの 0 始まりインデックス。これにより Claude が URL の転記ミスをしても記事の特定に影響しない。

`_parse_response()` では `re.search(r"\[.*\]", text, re.DOTALL)` で JSON 配列部分を抽出してパースするため、前後に説明文があっても対応できる。

### discord_notifier.py

**責務:** 厳選記事リストを Discord Webhook API に Embed 形式で送信する

**主要ロジック:**

- `_build_payload()`: 5件の Article を1つの Webhook リクエストにまとめる
  - `title[:256]`: Discord の Embed title 上限 256 文字でクリップ
  - `summary[:200]`: description を 200 文字でクリップ（5件 × 200 文字 = 1,000 文字で Discord 上限 6,000 文字に余裕）
- リトライ対象は 429（レート制限）と 5xx（サーバーエラー）のみ。400/401 は即時失敗

### lambda_function.py

**責務:** 4つのモジュールを順番に呼び出すオーケストレーター

```
get_config() → fetch_all_articles() → rank_and_summarize() → send()
```

各モジュールで発生した例外はそのまま上位に伝播させる。Lambda が失敗として記録し、CloudWatch Logs に `[ERROR]` ログが残る。

`if __name__ == "__main__":` ブロックにより、ローカルから直接 `python lambda_function.py` で実行できる。

## 4. Lambda Layerのビルド・デプロイ手順

### 4.1 構成概要

- **Lambda Layer**: feedparser, httpx, anthropic の3つの外部ライブラリ
- **Lambda 関数本体**: アプリコード（7つの .py ファイル）

### 4.2 Layer のビルド（Docker 使用）

Amazon Linux 2023 互換の環境でビルドする必要がある。

```bash
cd NewsBot/newsbot

# layer ディレクトリを作成
mkdir -p layer

# Docker で Amazon Linux 2023 互換環境にインストール
docker run --rm \
  -v $(pwd):/var/task \
  public.ecr.aws/lambda/python:3.13 \
  pip install -r /var/task/requirements.txt \
      -t /var/task/layer/python/lib/python3.13/site-packages/

# zip 化
cd layer
zip -r ../newsbot-layer.zip python/
cd ..
```

### 4.3 アプリコードの zip 化

```bash
cd NewsBot/newsbot

zip newsbot-app.zip \
  lambda_function.py \
  ssm_client.py \
  news_fetcher.py \
  ai_ranker.py \
  discord_notifier.py \
  models.py \
  config.py
```

### 4.4 Lambda Layer のアップロード

```bash
aws lambda publish-layer-version \
  --layer-name newsbot-dependencies \
  --zip-file fileb://newsbot-layer.zip \
  --compatible-runtimes python3.13 \
  --region ap-northeast-1
```

出力の `LayerVersionArn` を控えておく（例: `arn:aws:lambda:ap-northeast-1:123456789012:layer:newsbot-dependencies:1`）。

### 4.5 アプリコードのアップロード

```bash
aws lambda update-function-code \
  --function-name newsbot-delivery \
  --zip-file fileb://newsbot-app.zip \
  --region ap-northeast-1
```

### 4.6 Layer の紐付け

```bash
aws lambda update-function-configuration \
  --function-name newsbot-delivery \
  --layers arn:aws:lambda:ap-northeast-1:123456789012:layer:newsbot-dependencies:1 \
  --region ap-northeast-1
```

### 4.7 ランタイム設定の確認

Lambda コンソールで以下を確認する。

| 設定項目 | 値 |
|---------|---|
| ランタイム | Python 3.13 |
| ハンドラー | `lambda_function.lambda_handler` |
| タイムアウト | 5分（300秒）以上を推奨 |
| メモリ | 256 MB 以上を推奨 |

## 6. トラブルシューティング

### エラー: `ParameterNotFound` / `AccessDeniedException`

**原因:** SSM パラメータ名の誤り、または Lambda 実行ロールに SSM 権限がない

**対処:**
1. SSM パラメータ名が `/newsbot/anthropic_api_key`、`/newsbot/discord_webhook_url`、`/newsbot/discord_channel_name` であることを確認
2. Lambda 実行ロールに以下の権限があることを確認

```json
{
  "Action": "ssm:GetParameter",
  "Resource": "arn:aws:ssm:ap-northeast-1:*:parameter/newsbot/*"
}
```

### エラー: `ModuleNotFoundError: No module named 'feedparser'`

**原因:** Lambda Layer が正しくアタッチされていない

**対処:**
1. Lambda コンソールの「レイヤー」セクションで `newsbot-dependencies` がアタッチされているか確認
2. Layer のビルドが `python3.13` 向けであることを確認
3. Layer の zip 構造が `python/lib/python3.13/site-packages/` になっていることを確認

### エラー: `Claude response does not contain JSON array`

**原因:** Claude がシステムプロンプトの JSON 形式指示を無視して説明文を返した

**対処:**
1. CloudWatch Logs で Claude の実際のレスポンスを確認（`ValueError` の後に 200 文字が出力される）
2. システムプロンプトの指示が正しく設定されているか確認
3. 記事数が多すぎて max_tokens を超えていないか確認（config.py の `CLAUDE_MAX_TOKENS` を増やす）

### エラー: `[DISCORD] Failed after 3 retries`

**原因:** Discord Webhook URL が無効、またはレート制限が長時間続いている

**対処:**
1. Discord サーバーの設定から Webhook URL を再確認し、SSM の値を更新
2. Discord のステータスページでサービス障害がないか確認

### 全記事が0件になる

**原因:** 全ソースのネットワーク接続が失敗している（Lambda の VPC 設定、DNS 解決失敗など）

**対処:**
1. Lambda 関数が VPC 外（デフォルト設定）で実行されているか確認
2. CloudWatch Logs で `[FETCH]` の WARNING ログを確認し、失敗ソースと原因を特定

### ローカルテストで `ANTHROPIC_API_KEY` 環境変数エラー

**原因:** `LOCAL_TEST=1` 設定時に環境変数が設定されていない

**対処:**

```bash
export LOCAL_TEST=1
export ANTHROPIC_API_KEY="sk-ant-..."
export DISCORD_WEBHOOK_URL="https://discord.com/api/webhooks/..."
export DISCORD_CHANNEL_NAME="it-news"
```
